## 4. Acceptance Criteria (Must Pass)

### 4.1 Feature F-024: Transport Connection Interface (F-01)

#### Functional criteria

- [ ] **F-024-F01**: A `Connection` interface exists at `pkg/transport/connection.go` (or remains in `pkg/transport/quic/conn.go` as `quictr.Connection`) that all consumers reference. The interface must define: ID(), RemoteAddr(), Send(), SendControl(), SendProbe(), RecvData(), RecvControl(), RecvProbe(), RTT(), Stats(), Close(), IsClosed().
- [ ] **F-024-F02**: Pool, Arbitrator, Sender, Receiver, PathManager, and PoolManager all reference the Connection interface -- not a concrete QUIC type.

#### Tests

- [ ] **F-024-T01**: Existing tests continue to compile and pass after any refactoring.

### 4.2 Feature F-025: Config Externalisation (F-02)

#### Functional criteria

- [ ] **F-025-F01**: Hardcoded constants (`fragmentsPerBatch` in sender, `maxCommandsPerCycle` in conductor, `broadcastMaxPayload` in conductor, probe interval in pathmgr) are surfaced as configurable parameters in their respective structs or constructors.
- [ ] **F-025-F02**: Defaults match current hardcoded values (fragmentsPerBatch=32, maxCommandsPerCycle=10, broadcastMaxPayload=512).

#### Tests

- [ ] **F-025-T01**: Tests verify default values match legacy hardcoded values.
- [ ] **F-025-T02**: Tests verify custom values take effect.

### 4.3 Feature F-026: Broadcast Lapping Reconciliation (A-04)

#### Functional criteria

- [ ] **F-026-F01**: After `pollBroadcast` detects ErrLapped, the client re-queries the conductor's publication and subscription state to detect any missed `RspPublicationReady` or `RspSubscriptionReady` responses.
- [ ] **F-026-F02**: Pending requests that were fulfilled by the missed broadcast are resolved.

#### Tests

- [ ] **F-026-T01**: Test verifies that after lapping, pending publication requests are reconciled.

### 4.4 Feature F-027: Lock-Free Conductor Reads (P-03)

#### Functional criteria

- [ ] **F-027-F01**: `Conductor.Publications()` and `Conductor.Subscriptions()` use `atomic.Pointer` (or equivalent lock-free mechanism) for reads, eliminating mutex acquisition on the Sender/Receiver hot path.
- [ ] **F-027-F02**: Write paths (handleAddPublication, handleRemovePublication, etc.) still correctly publish updated snapshots.

#### Tests

- [ ] **F-027-T01**: Existing conductor tests pass without modification.
- [ ] **F-027-T02**: Concurrent read/write test demonstrates no data race (via -race flag).

### 4.5 Feature F-028: Sticky Arbitrator Pin Cleanup (C-08)

#### Functional criteria

- [ ] **F-028-F01**: Sticky arbitrator exposes a `Remove(publicationID int64)` method that deletes the pin entry.
- [ ] **F-028-F02**: The `Remove` method is called when a publication is removed (either wired from conductor or documented for future wiring).

#### Tests

- [ ] **F-028-T01**: Test verifies that after Remove, the next Pick for that publicationID re-pins to the best connection.
- [ ] **F-028-T02**: Test verifies that pins map does not grow unbounded after repeated add/remove cycles.

### 4.6 Feature F-029: Adaptive pollBroadcast Backoff (A-05)

#### Functional criteria

- [ ] **F-029-F01**: `pollBroadcast` uses adaptive backoff: starts at 100us, increases to a maximum (e.g., 1ms) after N consecutive idle cycles (e.g., 10), resets to 100us on any received message.
- [ ] **F-029-F02**: The adaptive backoff reduces CPU usage during idle periods compared to fixed 100us sleep.

#### Tests

- [ ] **F-029-T01**: Test verifies that idle polling increases sleep duration.
- [ ] **F-029-T02**: Test verifies that receiving a message resets backoff to minimum.

### 4.7 Non-Functional Criteria (All Features)

- [ ] **NF-01**: `go build ./...` exits 0
- [ ] **NF-02**: `go test -race ./...` exits 0
- [ ] **NF-03**: No regression in existing packages
