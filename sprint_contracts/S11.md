## 1. Sprint Description

Sprint S11 fixes the most critical production blockers in the Hyperspace hot path. The Receiver writes incomplete frame headers (C-05), causing readers to interpret frameType=0 (PAD) and silently drop ALL received data. Term rotation in Publication is non-atomic (C-01), allowing concurrent publishers to corrupt the log. The Sender allocates a new byte slice on every frame send (P-01), causing 1M allocs/sec. Agent goroutines have no panic recovery (A-01), meaning any panic kills the entire driver. The Receiver image map grows unbounded (S-01), leaking 48 MB per publisher with no eviction.

## 4. Acceptance Criteria (Must Pass)

All of the following must be true for the Evaluator to return PASS.

### 4.1 Feature F-015: Receiver Complete Frame Header (C-05)

#### Compilation and structure

- [ ] **F-015-C01**: `pkg/driver/receiver/receiver.go` compiles without errors
- [ ] **F-015-C02**: Frame write in `processFrame` writes ALL header fields: frameLength, version, flags, frameType, termOffset, sessionID, streamID, termID, reservedValue

#### Functional criteria

- [ ] **F-015-F01**: After a frame is written to the image log buffer, reading the header via `logbuffer.Header` returns the correct `FrameType` (FrameTypeDATA, not FrameTypePAD)
- [ ] **F-015-F02**: After a frame is written, `Header.SessionID()`, `Header.StreamID()`, `Header.TermID()`, `Header.Flags()`, `Header.Version()` all match the values from the incoming frame
- [ ] **F-015-F03**: The frameLength field is written LAST (volatile store) to signal readers -- write ordering must be preserved

#### Tests

- [ ] **F-015-T01**: `TestDoWork_WritesCompleteFrameHeader` -- receive a frame, read back ALL header fields from the image log buffer, assert each matches the input
- [ ] **F-015-T02**: `TestDoWork_FrameTypeIsDATA_NotPAD` -- receive a DATA frame, read back frameType from image, assert it is FrameTypeDATA (not FrameTypePAD/0)

### 4.2 Feature F-016: Atomic Term Rotation (C-01)

#### Compilation and structure

- [ ] **F-016-C01**: `pkg/client/publication.go` compiles without errors
- [ ] **F-016-C02**: Term rotation uses `CompareAndSwapInt32` (CAS loop) on ActivePartitionIndex, not `SetActivePartitionIndex`

#### Functional criteria

- [ ] **F-016-F01**: When two goroutines both observe `AppendRotation` simultaneously, only one succeeds in advancing the partition index -- the other retries from the new partition
- [ ] **F-016-F02**: No data corruption occurs under concurrent Offer from multiple goroutines (verified by race detector)

#### Tests

- [ ] **F-016-T01**: `TestOffer_ConcurrentRotation_NoCorruption` -- 4 goroutines offering simultaneously on a small-term publication, verify no data races and all messages are delivered or properly back-pressured
- [ ] **F-016-T02**: `TestOffer_CASRotation_AdvancesOnce` -- verify that concurrent AppendRotation results from multiple goroutines only advance the partition index by 1, not N

### 4.3 Feature F-017: Sender Frame Buffer Pool (P-01)

#### Compilation and structure

- [ ] **F-017-C01**: `pkg/driver/sender/sender.go` compiles without errors
- [ ] **F-017-C02**: A `sync.Pool` is used to provide reusable frame buffers instead of `make([]byte, frameLen)` on every send

#### Functional criteria

- [ ] **F-017-F01**: After warm-up (first batch), no new heap allocations occur in `sendPublication` for frames up to MTU size
- [ ] **F-017-F02**: Buffers are properly returned to the pool after use (no leaks)
- [ ] **F-017-F03**: The frame data written to the connection is identical with or without pooling (correctness preserved)

#### Tests

- [ ] **F-017-T01**: `TestDoWork_UsesBufferPool` -- send 100 frames, verify all are received correctly (correctness)
- [ ] **F-017-T02**: `BenchmarkSendPublication` -- benchmark shows zero allocs per send after warm-up

### 4.4 Feature F-018: Agent Panic Recovery (A-01)

#### Compilation and structure

- [ ] **F-018-C01**: `pkg/driver/agent.go` compiles without errors
- [ ] **F-018-C02**: `RunAgent` wraps each `DoWork` call in a deferred `recover()` that catches panics

#### Functional criteria

- [ ] **F-018-F01**: When `DoWork` panics, the agent goroutine does NOT terminate -- it logs the panic and continues the duty cycle
- [ ] **F-018-F02**: A panic counter is incremented atomically on each recovered panic
- [ ] **F-018-F03**: If panic count exceeds a configurable threshold (default 10), the agent stops gracefully and logs a fatal-level message
- [ ] **F-018-F04**: Non-panicking agents are unaffected by the recovery mechanism (zero overhead for the non-panic path)

#### Tests

- [ ] **F-018-T01**: `TestRunAgent_RecoversPanic` -- agent that panics once continues running after recovery
- [ ] **F-018-T02**: `TestRunAgent_PanicCounter_ExceedsThreshold_Stops` -- agent that panics N+1 times stops after threshold is reached
- [ ] **F-018-T03**: `TestRunAgent_NoPanic_NoOverhead` -- agent that never panics runs normally with zero panic counter

### 4.5 Feature F-019: Image Map TTL Eviction (S-01)

#### Compilation and structure

- [ ] **F-019-C01**: `pkg/driver/receiver/receiver.go` compiles without errors
- [ ] **F-019-C02**: Image map entries have an associated `lastAccess` timestamp

#### Functional criteria

- [ ] **F-019-F01**: Image log buffers that have not been accessed for TTL duration (default 60s) are evicted from the map
- [ ] **F-019-F02**: Eviction is triggered periodically (e.g., every N DoWork calls or on a timer), not on every frame receive
- [ ] **F-019-F03**: Active images (recently accessed) are NOT evicted
- [ ] **F-019-F04**: When `CmdRemoveSubscription` is processed, the corresponding image is removed immediately (not deferred to TTL)

#### Tests

- [ ] **F-019-T01**: `TestEviction_StaleSessions_AreRemoved` -- create images, advance time past TTL, verify they are evicted
- [ ] **F-019-T02**: `TestEviction_ActiveSessions_AreKept` -- create images, keep accessing them, verify they survive past TTL
- [ ] **F-019-T03**: `TestEviction_NewFrameUpdatesLastAccess` -- receive a frame, verify lastAccess is updated
