The Remediation Report identified 12 violations across 22 requirements (10 violated, 2 partially followed that represent process failures). Root cause analysis traced every violation to three systemic failures: (1) the sprint map deferred all documentation to S10 with no per-sprint documentation contract, (2) the Harness Engineering Team ran only once at kickoff and was never re-invoked at sprint boundaries, and (3) no enforcement mechanism exists — agents could proceed without sprint contracts, without Evaluator PASS verdicts, and without updating session artefacts.

This report identifies 9 structural gaps in the framework — missing files, missing enforcement language in agent prompts, missing CI checks, and missing gate artefacts — that, if implemented, would make it structurally impossible for these violations to recur. The gaps are ranked by enforcement strength: P0 gaps prevent incorrect work from starting, P1 gaps detect deviation during execution, and P2 gaps improve long-term quality.

Gap Analysis

P0 GAP-1: No init.sh Session Initialisation Script
Would have prevented: Session protocol violations (session_state.json not updated, harness_telemetry.jsonl not maintained, baseline not verified before new work)
Root cause addressed: #3 — No enforcement mechanism between sprint boundaries
Artefact needed: init.sh at project root (executable shell script)
Required content: The script must:
1. Verify session_state.json exists; create if first session
2. Verify progress.json exists; exit 1 if missing
3. Verify the current sprint has a signed sprint contract in sprint_contracts/; exit 1 if missing
4. Verify HARNESS_QUALITY_REPORT.md exists and contains a PASS or CONDITIONAL PASS verdict; exit 1 if FAIL or missing
5. Run make test and exit 1 if tests fail (baseline must be green)
6. Print a structured JSON summary to stdout with fields: session_ok, sprint_id, baseline_green, harness_verdict
7. Exit 0 only if all checks pass

The Engineering Orchestrator and all implementing agents must run ./init.sh as the first action of every session. If it exits non-zero, no work begins. This script was identified in HARNESS_IMPROVEMENT_REPORT.md Recommendation 6 and in Principle 2.5 of HARNESS_ENGINEERING_PRINCIPLES.md, but was never created.
P0 GAP-2: No Sprint Gate Template (SPRINT_GATE.md)
Would have prevented: Sprint contracts not produced for S3–S10; Harness Evaluator gate skipped
Root cause addressed: #2 — Harness Engineering Team ran only once
Artefact needed: sprint_contracts/SPRINT_GATE_TEMPLATE.md (mandatory template)
Required content: A template that the Harness Evaluator must fill and sign before any sprint begins. Fields:
sprint_id: [S-ID]
sprint_contract_exists: [YES/NO] — path to sprint contract file
harness_evaluator_verdict: [PASS / CONDITIONAL PASS / FAIL]
harness_quality_report_date: [ISO date of most recent HARNESS_QUALITY_REPORT.md]
documentation_deliverables: [list of docs this sprint must produce]
evaluator_assigned: [YES/NO] — Code Evaluator assigned for end-of-sprint review
signed_by: Harness Orchestrator
date: [ISO date]

Each sprint contract (e.g. sprint_contracts/S3.md) must include a documentation section listing what docs, knowledge base entries, and session artefacts that sprint must produce. No sprint begins without a completed SPRINT_GATE entry at the top of the sprint contract.
P0 GAP-3: No HARD STOPS in Agent Prompts
Would have prevented: All 12 violations — agents proceeded without contracts, without evaluator gates, without updating session artefacts
Root cause addressed: #3 — No enforcement mechanism
Artefact needed: Updates to CLAUDE.md (project root) and all agent prompt files in .claude/agents/
Required content: Every agent prompt must include a HARD STOPS section. The project CLAUDE.md must include a top-level HARD STOPS section (exact text provided in Section 6 of this report). Each implementing agent (backend.md, frontend.md, etc.) must include:

## HARD STOPS — I Will Not Proceed If:
1. No sprint contract exists for the current sprint in sprint_contracts/
2. HARNESS_QUALITY_REPORT.md does not contain PASS or CONDITIONAL PASS
3. make test fails (baseline is not green)
4. session_state.json does not exist
5. I am asked to mark a feature as passing without Code Evaluator PASS verdict

The Engineering Orchestrator prompt must additionally include:
6. I will not assign implementation work without a signed sprint contract
7. I will invoke the Harness Evaluator at every sprint boundary — no exceptions
8. I will not deliver to the user without Code Evaluator PASS
P0 GAP-4: progress.json Tracks Sprints, Not Features
Would have prevented: Feature-level tracking violation; inability to identify which features passed evaluation
Root cause addressed: #3 — No enforcement mechanism (no granular tracking to gate against)
Artefact needed: progress.json schema definition + replacement of current file
Current state: progress.json tracks 10 sprints with status "complete". It does not track the 14 individual features (F-001 through F-014).
Required schema:
{ "project": "hyperspace", "last_updated": "2026-04-18", "features": [ { "id": "F-001", "name": "Log Buffer", "sprint": "S1", "status": "passing | failing | not_started | in_progress", "evaluator_verdict": "PASS | FAIL | NOT_EVALUATED", "evaluator_date": "ISO date or null", "satisfaction_score": 0.0-1.0, "blocked_by": ["F-ID"] or [] } ] }
The evaluator_verdict field is the machine-checkable gate: no feature may have status: "passing" unless evaluator_verdict: "PASS". The init.sh script validates this constraint.
P1 GAP-5: No CI Check for Harness Artefact Existence
Would have prevented: Missing HARNESS_QUALITY_REPORT.md, missing sprint contracts, missing knowledge base entries
Root cause addressed: #2 and #3 — No re-invocation and no enforcement
Artefact needed: New job in .github/workflows/ci.yml
Required content: A harness-check job that runs on every PR and verifies:
1. HARNESS_QUALITY_REPORT.md exists and was updated within the last 14 days
2. progress.json contains feature-level entries (not sprint-level)
3. session_state.json was updated within the last 7 days
4. For every sprint directory sprint_contracts/S*.md, the file is non-empty

This job fails the PR if any check fails. It provides the automated enforcement layer that agent discipline alone cannot guarantee.

P1 GAP-7: No Scenario Holdout Tests
Would have prevented: Self-evaluation loops; implementing agents declaring their own work complete
Root cause addressed: #2 — Harness Engineering Team ran only once (scenarios never authored)
Artefact needed: scenarios/ directory with actual scenario files + scenarios/runner.py
Current state: The scenarios/ directory contains only a README.md. No actual scenario files exist. Principle 1.2 of HARNESS_ENGINEERING_PRINCIPLES.md explicitly states that holdout scenarios are the correctness mechanism and that a harness without them is "structurally equivalent to a machine learning model evaluated on its own training data."

Required:
1. At minimum 3 scenario files per sprint, authored by the Harness Orchestrator (not the implementing agent)
2. A scenarios/runner.py script that executes scenarios and produces a satisfaction score
3. Scenarios must be read-only to implementing agents
4. The Evaluator runs scenarios as part of the sprint-end evaluation
5. Satisfaction score >= 0.95 required for PASS; 0.80-0.95 for CONDITIONAL PASS; below 0.80 is FAIL

New Agent Required: Business Requirements Agent

The current framework assumes that a well-formed project brief arrives at the Engineering Orchestrator. In practice, user requests are often ambiguous, incomplete, or describe a solution rather than a problem. A Business Requirements Agent should exist as the first stage in the pipeline, before the CTO/Engineering Orchestrator classifies and routes work.

Agent Definition

Name: Business Requirements Agent
Model: claude-opus-4-7 (requires deep reasoning about business context)
Reports to: CTO / Engineering Orchestrator
Triggers: Every new project request, every major scope change

Responsibilities:
1. Receive raw user requests (natural language, product briefs, competitor references, customer feedback)
2. Elicit missing requirements through structured questioning
3. Classify requirements: functional, non-functional, constraints, out-of-scope
4. Identify stakeholders and their priorities
5. Produce SPEC.md using a structured template with acceptance criteria
6. Identify technical risks and unknowns that require CTO decision
7. Produce a BUSINESS_REQUIREMENTS.md that maps business goals to technical requirements
8. Hand off to CTO for approval, then to Harness Architect for system design

Mandatory outputs:
• BUSINESS_REQUIREMENTS.md — business context, user stories, success metrics, constraints
• SPEC.md (draft) — functional and non-functional requirements with acceptance criteria
• Risk register — technical unknowns and business risks requiring CTO decision

Quality gate: CTO must approve BUSINESS_REQUIREMENTS.md before the Harness Architect begins SYSTEM_ARCHITECTURE.md. This ensures that the architecture is driven by validated business requirements, not by the implementing agent's assumptions.

Why This Agent Is Needed

In the Hyperspace project, the Harness Architect produced SYSTEM_ARCHITECTURE.md and SPEC.md directly from a technical brief. There was no stage where business requirements were validated, prioritised, or checked against user needs. The sprint map deferred documentation to S10 partly because there was no business-level requirement for incremental documentation — no one had specified "the user needs documentation at each milestone" as a requirement. A Business Requirements Agent would have caught this gap at the requirements stage, before any architecture or sprint planning occurred.

Recommended Changes to CLAUDE.md

The following three sections should be added to the project CLAUDE.md (at the project root). They should be placed immediately after the "Critical Rules" section and before the "Session Protocol" section, making them the first thing any agent reads after the non-negotiable rules.

Addition 1: HARD STOPS Section

## HARD STOPS (Non-Negotiable Agent Refusals) **Every agent operating in this repository MUST refuse to proceed if any of the following conditions are true.** These are not guidelines. These are blocking conditions. An agent that proceeds despite a HARD STOP is in violation of the framework. ### All Implementing Agents (Backend, Frontend, DevOps, Security, QA, AI/ML, Data) 1. **No sprint contract exists.** If `sprint_contracts/S[current].md` does not exist for the current sprint, STOP. Do not begin implementation. Escalate to the Engineering Orchestrator. 2. **Harness Evaluator has not issued PASS.** If `HARNESS_QUALITY_REPORT.md` does not exist or contains a FAIL verdict, STOP. Do not begin implementation. Escalate to the Engineering Orchestrator. 3. **Baseline tests are not green.** If `make test` fails, STOP. Diagnose and fix the failure before any new work. Do not implement new features on a broken baseline. 4. **Session state file is missing.** If `session_state.json` does not exist and this is not the first session, STOP. The project state is unknown. Escalate. 5. **Feature already marked passing without Evaluator PASS.** If `progress.json` shows a feature as `passing` but `evaluator_verdict` is not `PASS`, do not treat it as complete. Report the inconsistency. 6. **Sprint contract has no documentation deliverables.** If the sprint contract does not include a "Documentation Deliverables" section, STOP. The contract is incomplete. Escalate to the Harness Architect. ### Engineering Orchestrator (Additional) 7. **Do not assign implementation work without a signed sprint contract.** The sprint contract must exist in `sprint_contracts/` with a completed SPRINT_GATE section before any implementing agent receives a task. 8. **Invoke the Harness Evaluator at every sprint boundary.** At the end of every sprint, before starting the next, the Harness Evaluator must produce or update `HARNESS_QUALITY_REPORT.md`. This is not optional. This is not "when convenient." This is every sprint. 9. **Do not deliver to the user without Code Evaluator PASS.** The Code Evaluator must independently verify the implementation before the Engineering Orchestrator approves delivery. 10. **Do not skip the Business Requirements stage for new projects.** Every new project must have a validated `BUSINESS_REQUIREMENTS.md` (or equivalent approved brief from CTO) before the Harness Architect begins.

Addition 2: PRE-SPRINT GATE Section

## PRE-SPRINT GATE (Must Be Verified Before Any Sprint Begins) Before the Engineering Orchestrator assigns any implementation work for sprint S[N], the following must be true: ### Mandatory Pre-Sprint Checklist - [ ] **Sprint contract exists:** `sprint_contracts/S[N].md` is present, non-empty, and follows the template - [ ] **Documentation deliverables listed:** The sprint contract includes a "Documentation Deliverables" section with at least one deliverable - [ ] **HARNESS_QUALITY_REPORT.md is current:** The report was updated within the last sprint cycle and contains PASS or CONDITIONAL PASS - [ ] **progress.json is feature-level:** Every feature (F-001 through F-014) has an entry with `status`, `evaluator_verdict`, and `satisfaction_score` fields - [ ] **Previous sprint evaluated:** The Code Evaluator has issued a verdict for every feature in the previous sprint that was marked `in_progress` or `passing` - [ ] **Baseline tests green:** `make test` exits 0 on the current main branch - [ ] **Session artefacts current:** `session_state.json` and `session_handoff.md` were updated at the end of the previous sprint ### If Any Item Fails The Engineering Orchestrator MUST NOT assign implementation work. Instead: 1. Invoke the Harness Evaluator to assess the gap 2. Invoke the relevant harness team member to fix the gap 3. Re-verify the checklist 4. Only proceed when all items are checked
